# Wordrill 제품 기획 메모

이 문서는 제품 방향, 미결정 사항, 기능별 위험과 의사결정을 지속적으로 기록한다.

## 구상 중인 채팅방 유형

### 공개 채팅방

- 로그인 없이 게스트가 가볍게 입장할 수 있다.
- 공개 목록 또는 공유 가능한 URL을 통해 접근한다.
- 게스트 로그인/입장을 가장 먼저 구현한다.

### 회원 채팅방

- 일반적인 채팅 앱처럼 회원이 방을 만들고 참여할 수 있다.
- 현재의 `User`, `RoomMember`, `Room` 구조에 가장 가깝다.

### 비밀 채팅방

- 초대받은 사용자만 방의 존재와 내용을 보고 입장할 수 있다.
- 초대 토큰, 초대 대상, 만료 및 철회 정책이 필요하다.

### 임시 채팅방

- 정해진 기간 또는 시각까지만 메시지 전송·수정·삭제가 가능하다.
- 마감 이후에는 읽기만 가능하다.
- 방의 만료와 메시지별 수정/삭제 기한 중 어떤 모델을 사용할지는 미정이다.

## 게스트 공개 채팅방 검토

### 현재 구조와의 차이

현재 시스템은 NextAuth 세션으로 DB의 `User`를 찾고, `RoomMember` 가입 레코드를 만든 뒤에만 소켓 채팅에 입장시킨다. 메시지는 `Message.userId`로 회원과 연결된다. 게스트 입장을 제공하려면 회원과 다른 단기 신원, 공개방 접근 정책, 게스트 메시지의 소유권 및 남용 방지 정책이 필요하다.

### 예상 문제점

#### 1. 신원 위조와 메시지 소유권

- 게스트는 영구 계정이 없으므로 재접속한 사람을 같은 사람으로 판단하기 어렵다.
- 닉네임만으로 구분하면 다른 사용자가 같은 닉네임을 사용하거나 사칭할 수 있다.
- 브라우저 저장소만 사용하면 저장소 삭제, 시크릿 모드, 다른 기기에서 신원이 사라진다.
- 메시지 수정·삭제를 지원할 때 작성자임을 안전하게 증명할 방법이 필요하다.
- 현재 소켓 서버는 클라이언트가 보낸 `userId` 또는 `email`로 사용자를 찾지만 NextAuth 세션과 대조하지 않는다. 게스트 기능과 별개로 기존 회원 사칭 가능성을 막기 위해 서버가 검증한 세션 또는 서명된 토큰을 사용해야 한다.

#### 2. 스팸, 도배 및 자동화 공격

- 계정 생성 장벽이 없어서 메시지 도배, 광고, 봇 접속이 쉬워진다.
- 재접속만으로 제재를 회피할 수 있다.
- 접속·입장·메시지 전송에 IP 및 게스트 단위 rate limit이 필요하다.
- 메시지 길이 제한만으로는 연속 전송, 다중 소켓, 반복 재입장을 막을 수 없다.
- 공개 서비스 단계에서는 CAPTCHA 또는 유사한 봇 방어가 필요할 수 있다.

#### 3. 신고, 차단 및 운영 대응

- 운영자가 게스트를 음소거하거나 차단할 안정적인 식별자가 필요하다.
- IP 영구 차단은 공유망 사용자를 함께 차단할 수 있고 개인정보 취급 부담도 생긴다.
- 신고 대상 메시지와 당시 작성자 식별 정보를 일정 기간 보존할 정책이 필요하다.
- 금칙어, 링크 제한, 신고, 관리자 삭제, 임시 차단의 최소 운영 도구가 필요하다.

#### 4. 닉네임과 사용자 경험

- 닉네임 중복 허용 여부와 방 안에서의 표시 방식이 필요하다.
- 회원과 게스트를 시각적으로 구별하지 않으면 신뢰 문제가 생길 수 있다.
- 게스트 닉네임 변경 시 과거 메시지의 표시 이름을 유지할지 변경할지 결정해야 한다.
- 새로고침 또는 재접속 시 닉네임과 작성 메시지 소유권을 얼마나 복원할지 정해야 한다.

#### 5. 공개방의 노출 범위

- 공개방이 URL을 아는 사람만 들어오는 방인지, 공개 목록과 검색에 노출되는 방인지 구분해야 한다.
- 검색 엔진 색인 허용 여부를 결정해야 한다.
- 방 목록과 메시지 기록을 로그인 없이 조회할 수 있다면 크롤링과 대량 수집에 대비해야 한다.
- 공개방 생성까지 게스트에게 허용하면 스팸 방과 부적절한 방 이름이 급증할 수 있다.

#### 6. 데이터 모델과 데이터 수명

- 모든 게스트를 `User` 행으로 만들면 일회성 계정 데이터가 계속 쌓인다.
- `User`를 만들지 않으면 현재 `RoomMember`와 작성자 관계를 그대로 쓰기 어렵다.
- 게스트 세션 만료 후에도 메시지를 보존할지, 작성자 식별 정보를 익명화할지 결정해야 한다.
- 탈퇴 개념이 없는 게스트의 개인정보 삭제 요청과 보존 정책을 정의해야 한다.

#### 7. 권한 경계 복잡성

- 공개방은 게스트 입장을 허용하지만 회원방과 비밀방은 거부해야 한다.
- 방 조회, 입장, 기록 읽기, 메시지 쓰기를 각각 독립된 권한으로 검사해야 한다.
- 화면에서 버튼을 숨기는 것만으로는 부족하며 HTTP API와 Socket.IO 이벤트 모두 서버에서 검사해야 한다.
- 향후 임시방의 읽기 전용 전환까지 추가되면 방 유형과 현재 상태를 함께 평가하는 공통 권한 계층이 필요하다.

#### 8. 동시 접속과 비용

- 무인증 공개방은 예상보다 많은 소켓 연결과 재연결을 유발할 수 있다.
- 현재 presence 정보는 서버 메모리에 있어 다중 서버로 확장하면 서버 간 동기화가 되지 않는다.
- 연결 수, 방별 인원, 메시지 빈도 제한과 관측 지표가 필요하다.

#### 9. 보안과 개인정보

- XSS를 막기 위해 메시지와 닉네임을 계속 일반 텍스트로 렌더링하고, 향후 링크·마크다운 도입 시 별도 정제가 필요하다.
- IP, User-Agent, 기기 식별값을 저장한다면 수집 목적과 보존 기간을 정하고 최소화해야 한다.
- 공개 메시지는 누구나 볼 수 있다는 사실을 입장 전에 명확히 안내해야 한다.

### 게스트 MVP 권장 범위

첫 구현은 다음처럼 제한하는 것이 안전하다.

- 게스트는 운영자가 미리 만든 `PUBLIC` 방만 URL로 입장할 수 있다.
- 게스트의 방 생성, 회원방 접근, 비밀방 접근은 허용하지 않는다.
- 서버가 임의의 `guestSessionId`와 서명된 단기 토큰을 발급한다.
- 브라우저에는 원본 권한을 만들 수 없는 서명된 토큰만 저장한다.
- 닉네임은 방 입장 시 입력하며 `Guest` 표시를 붙인다.
- 게스트 세션에는 만료 시간을 둔다. 재접속 복원 범위는 같은 브라우저와 만료 전으로 제한한다.
- 메시지 전송에 게스트·IP·소켓 단위 rate limit과 최대 길이를 적용한다.
- MVP에서는 메시지 수정·삭제를 제공하지 않거나, 유효한 게스트 세션에서 짧은 시간 동안만 허용한다.
- 방별 최대 동시 접속자와 최소한의 관리자 음소거/차단 수단을 둔다.
- HTTP와 Socket.IO가 같은 권한 판정 함수를 사용하도록 구성한다.

### 구현 전에 결정할 항목

- 공개방은 공개 목록에 노출할지, URL 접근만 허용할지
- 게스트가 과거 메시지를 어느 범위까지 볼 수 있는지
- 게스트 닉네임 중복을 허용할지
- 게스트 세션의 유효 기간
- 게스트 메시지의 보존 기간과 수정·삭제 허용 여부
- 공개방을 만들 수 있는 주체: 운영자만 또는 로그인 회원
- 신고 및 차단 기능을 게스트 MVP에 포함할지

### 우선순위

1. 서버가 검증하는 회원/게스트 신원 체계
2. `PUBLIC`, `MEMBER`, `PRIVATE`, `TEMPORARY`를 표현할 방 접근 모델
3. HTTP 및 Socket.IO 공통 권한 검사
4. 게스트 공개방 입장 UI와 닉네임 설정
5. rate limit, 신고 및 운영 도구
6. 임시방의 마감 및 읽기 전용 상태 모델

## 게스트 계정 수명주기와 회원 전환

### 권장 원칙: 데이터 이전보다 동일 계정 업그레이드

Firebase 익명 인증의 계정 연결과 유사하게, 게스트가 가입할 때 새 `User`를 만들고 데이터를 옮기기보다 기존 게스트 `User` 행에 이메일·Google 계정·비밀번호 자격 증명을 연결한다. 그러면 `User.id`가 유지되므로 `Message`, `RoomMember` 등 기존 관계를 마이그레이션할 필요가 없고 메시지 소유권도 그대로 유지된다.

게스트와 회원은 별도 테이블보다 하나의 `User` 모델에서 상태로 구분하는 방향을 우선 검토한다.

- `accountType`: `GUEST` 또는 `MEMBER`
- `status`: `ACTIVE`, `MERGED`, `DEACTIVATED`, `PENDING_DELETE` 등
- `lastSeenAt`: 실제 활동이 확인된 마지막 시각
- `guestExpiresAt`: 게스트 신원 만료 예정 시각
- `convertedAt`: 회원으로 전환된 시각
- `deletedAt`: 비활성화 또는 익명화된 시각

### 회원 전환 흐름

1. 서버가 서명된 게스트 세션으로 현재 게스트 `User.id`를 검증한다.
2. 이메일 또는 OAuth 자격 증명의 기존 사용 여부를 확인한다.
3. 기존 회원이 없다면 트랜잭션 안에서 같은 `User` 행에 인증 정보를 연결하고 `accountType`을 `MEMBER`로 변경한다.
4. 게스트 세션을 폐기하고 새 회원 세션을 발급한다.
5. 기존 메시지와 방 가입 관계는 같은 `User.id`를 계속 바라보므로 별도 이동하지 않는다.

이미 동일한 이메일 또는 OAuth 계정이 존재하는 경우는 계정 업그레이드가 아니라 계정 병합 문제다. 이때는 다음을 한 트랜잭션에서 처리한다.

1. 게스트의 `RoomMember`를 기존 회원 쪽으로 중복 없이 upsert한다.
2. 이전을 허용할 데이터만 기존 회원 ID로 변경한다.
3. 게스트 행에 `mergedIntoUserId`를 기록하고 `MERGED`로 비활성화한다.
4. 게스트 세션을 폐기하고 기존 회원 세션을 발급한다.

공개방의 단순 메시지는 작성 당시 닉네임을 보존하고 작성자 ID만 회원에게 연결할 수 있다. 신고·차단 기록, 제재 이력처럼 운영상 의미가 달라질 수 있는 데이터는 자동 병합 범위를 별도로 정해야 한다.

### 휴면 게스트 정리 정책

게스트 삭제는 `createdAt`이 아니라 `lastSeenAt`을 기준으로 한다. 발급만 받고 아무 활동이 없는 게스트와 실제 메시지를 남긴 게스트의 보존 기간을 다르게 두는 것이 좋다.

- 데이터가 없는 게스트: 마지막 활동 후 7일~30일 뒤 삭제 후보
- 메시지나 신고 관련 데이터가 있는 게스트: 30일~90일 뒤 계정 식별 정보 익명화
- 회원으로 전환된 계정: 게스트 정리 대상에서 제외
- `MERGED`, `PENDING_DELETE` 계정: 짧은 유예 기간 후 삭제 또는 익명화

정리 작업은 한 번에 모두 삭제하지 않고 작은 배치로 처리한다.

1. `accountType = GUEST`, `status = ACTIVE`, `lastSeenAt < cutoff`인 후보를 인덱스로 조회한다.
2. 최근 활성 세션, 보존 의무가 있는 신고·제재 데이터, 진행 중인 병합이 없는지 다시 확인한다.
3. 먼저 `PENDING_DELETE`로 표시하고 유예 기간을 둔다.
4. 다음 실행에서 여전히 비활성 상태인 계정만 삭제하거나 개인정보를 익명화한다.
5. 처리 건수와 실패를 기록해 재시도할 수 있게 한다.

현재 스키마에서 `User`를 삭제하면 `RoomMember`는 cascade 삭제되고 `Message.userId`는 `SetNull`이 된다. 메시지에는 작성 당시 `nickname`이 남으므로 공개 대화 기록을 유지하면서 게스트 신원만 제거할 수 있다. 메시지까지 삭제할지는 별도 보존 정책으로 결정한다.

### 스케줄러 운영 방안

정리 로직은 재사용 가능한 단독 스크립트(예: `scripts/cleanup-stale-guests.ts`)로 만들고 실행 주체만 환경에 따라 바꾼다.

- 로컬 개발: 수동 실행 또는 Windows 작업 스케줄러로 하루 1회 실행
- 단일 상시 Node 서버: 서버 내부 cron도 가능하지만 중복 등록과 서버 중단을 고려해야 함
- 배포 환경: 플랫폼 Cron이 보호된 내부 API 또는 정리 스크립트를 호출
- PostgreSQL 운영 환경: 필요하면 `pg_cron` 같은 DB 스케줄링도 검토

작업은 중복 실행되어도 결과가 같도록 만들고, DB advisory lock 또는 작업 잠금 행으로 동시에 두 번 실행되는 것을 막는다. 삭제 기준 시각은 애플리케이션 메모리가 아니라 DB의 UTC 시각을 기준으로 한다.

### 권장 초기값

- `lastSeenAt`은 인증된 HTTP 요청이나 소켓 활동에서 일정 간격으로 갱신한다. 메시지마다 쓰지 않고 예를 들어 5~15분 단위로 제한한다.
- 활동 없는 빈 게스트는 14일 후 `PENDING_DELETE`, 7일 유예 후 실제 삭제한다.
- 메시지를 남긴 게스트는 90일 후 계정 식별 정보만 익명화하고 메시지는 방 보존 정책을 따른다.
- 정리 배치는 100~500개 단위로 제한한다.
- 실제 삭제 전에 dry-run 모드로 후보 수와 영향을 확인할 수 있게 한다.

## 채팅방 멤버십 이벤트와 메시지 읽음 표시

소켓 연결 상태와 실제 채팅방 멤버십은 서로 다른 개념으로 취급한다. 페이지 새로고침, 브라우저 종료, 네트워크 단절은 온라인 상태만 변경하며 채팅 내 `joined` 또는 `left` 시스템 메시지를 생성하지 않는다.

- `joined`: 사용자의 `RoomMember`가 최초 생성된 경우에만 생성한다.
- `left`: 사용자가 명시적인 채팅방 나가기 기능을 실행한 경우에만 생성한다.
- 온라인/오프라인: 서버의 presence 정보와 멤버 목록 상태에만 반영한다.
- 초기 단계에서는 멤버십 이벤트 저장 모델이 정해질 때까지 `joined/left` 채팅 메시지를 숨기는 방향을 우선 적용한다.

카카오톡과 같은 메시지별 미확인 인원 수는 단순 온라인 인원이 아니라 각 멤버의 마지막 읽음 위치를 기준으로 계산한다. MVP에서는 `RoomMember.lastReadAt`을 추가하고, 화면에 실제로 렌더링된 최신 메시지를 사용자가 보고 있을 때 서버에 읽음 위치를 전송한다.

1. 메시지 전송 시 발신자를 제외한 방 멤버를 잠재적 미확인 인원으로 본다.
2. 사용자가 방을 열고 문서가 활성 상태이며 메시지가 화면에 표시되면 `message:read` 이벤트를 전송한다.
3. 서버는 인증된 멤버의 `lastReadAt`을 이전 값보다 뒤로만 이동시키고 `read:update`를 방에 브로드캐스트한다.
4. 각 메시지의 미확인 수는 `lastReadAt < message.createdAt`인 대상 멤버 수로 계산한다.
5. 미확인 수가 0이면 숫자를 숨긴다.

명시적인 방 나가기와 재가입까지 지원할 때는 `RoomMember`를 즉시 삭제하지 않고 `joinedAt`, `leftAt`으로 활동 기간을 남겨 메시지 발송 시점의 수신 대상이 누구였는지 판별한다. 메시지마다 모든 멤버의 읽음 행을 생성하는 `MessageRead` 방식은 정확하지만 데이터가 메시지 수와 멤버 수의 곱으로 증가하므로, 우선은 멤버별 마지막 읽음 위치 방식을 채택한다.

## UI 디자인 규칙

- 채팅방 이미지와 대체 아바타는 모서리가 둥근 정사각형으로 표시한다.
- 사용자 프로필 이미지와 대체 아바타는 원형으로 표시한다.
