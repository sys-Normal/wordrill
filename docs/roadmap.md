# Wordrill 로드맵

이 문서는 대화 기록에 의존하지 않고 다른 작업 세션에서도 프로젝트 방향과 진행 상태를 이어가기 위한 기준 문서다. 계획을 추가하거나 완료·보류·변경할 때 이 문서를 함께 갱신한다.

마지막 갱신: 2026-07-14

## 목표

기능 범위는 소규모 MVP로 유지하되, 인증·데이터 정합성·장애 복구·배포 구조는 실제 서비스에 사용할 수 있는 수준으로 준비한다.

웹과 향후 모바일 앱은 하나의 백엔드와 PostgreSQL 데이터베이스를 사용한다. 모바일 프로젝트를 만들 때 서버와 DB를 복제하지 않는다.

## 목표 구조

```text
웹 클라이언트 ─┐
모바일 앱 ─────┼─ API·Socket.IO 서버 ─ PostgreSQL
               │                    ├ Redis
관리 화면 ─────┘                    └ 객체 스토리지
```

- PostgreSQL: 사용자, 채팅방, 멤버, 메시지, 멘션, 읽음 위치의 기준 저장소
- Redis: 다중 서버 Socket.IO 이벤트 전달, 접속 상태, 짧은 수명의 임시 데이터
- 객체 스토리지: 향후 이미지와 첨부 파일
- 웹·모바일: 동일한 인증, HTTP API, Socket.IO 이벤트 계약 사용

## 진행 상태

### 1. PostgreSQL 메시지 저장 안정화 — 완료

- 커서 기반 과거 메시지 페이지네이션
- 클라이언트 메시지 ID와 고유 제약을 이용한 중복 저장 방지
- DB 메시지 순번을 이용한 안정적인 정렬
- 멤버별 마지막 읽은 메시지 순번 저장
- 읽지 않은 메시지·멘션 수 집계 쿼리 최적화
- 메시지 보존 dry-run 및 실행 스크립트
- 커넥션 풀, 백업·복구, 보존, 성장 모니터링 절차 문서화
- Prisma 마이그레이션, 타입 검사, 프로덕션 빌드, Socket.IO 통합 테스트 완료

관련 문서: [`database-operations.md`](database-operations.md)

### 2. Socket 인증과 권한 검증 강화 — 다음 작업

- `user:join` 등 Socket.IO 이벤트에서 클라이언트가 전달한 `userId`나 이메일을 사용자 신원의 근거로 신뢰하지 않는다.
- 서버 세션 또는 짧은 수명의 서명된 Socket 티켓으로 연결 사용자를 확정한다.
- 방 입장, 기록 읽기, 메시지 쓰기, 읽음 갱신 권한을 서버에서 각각 검사한다.
- HTTP API와 Socket.IO가 같은 공용 권한 판정 로직을 사용하게 한다.
- 인증 실패, 만료, 권한 부족에 대한 이벤트 오류 형식을 통일한다.

관련 기획: [`product-plan.md`](product-plan.md)

### 3. Redis 도입 — 대기

- PostgreSQL을 교체하지 않고 별도 인프라로 추가한다.
- Socket.IO Redis Adapter를 적용해 여러 서버 사이의 방 이벤트를 전달한다.
- 현재 프로세스 메모리의 온라인 사용자와 방 presence 상태를 Redis 기반으로 전환한다.
- Redis 장애 시 메시지 영구 데이터는 PostgreSQL에서 복구할 수 있어야 한다.
- 단일 서버 MVP에서도 적용 가능한 구성을 만들되, Redis 장애가 메시지 저장을 훼손하지 않도록 역할을 분리한다.

### 4. 공용 API·이벤트 타입 분리 — 대기

- 웹과 모바일이 공유할 HTTP 요청·응답 및 Socket.IO 이벤트 타입을 별도 패키지로 분리한다.
- 런타임 입력 검증 스키마와 TypeScript 타입이 같은 계약을 사용하게 한다.
- 서버 내부 Prisma 타입을 외부 API 계약으로 직접 노출하지 않는다.
- API 및 이벤트의 호환성 변경 원칙과 버전 정책을 정한다.

### 5. 배포·백업·로그 구성 — 일부 완료

완료:

- PostgreSQL 백업·복구 원칙과 메시지 보존 절차 문서화
- `/health` 상태 확인 엔드포인트

남은 작업:

- HTTPS/WSS 운영 배포 구성
- 환경변수와 비밀 정보 관리
- PostgreSQL 자동 백업 일정 및 복구 훈련
- 구조화된 애플리케이션 로그와 요청·소켓 상관관계 ID
- 오류 추적, 기본 메트릭, 알림
- Redis를 도입한 뒤 Redis 상태 점검 및 장애 대응 절차

### 6. 웹 MVP 버전 확정 — 대기

- MVP에 포함할 방 유형과 게스트 채팅 범위를 확정한다.
- 메시지 속도 제한, 입력 검증, 최소 신고·차단 정책을 반영한다.
- 핵심 사용자 흐름의 통합 테스트를 확정한다.
- 운영 배포와 복구 절차를 검증한 뒤 웹 MVP 버전을 태그한다.
- MVP 이후 기능은 별도 후속 단계로 이동한다.

### 7. 같은 서버를 사용하는 모바일 앱 개발 — 대기

- 웹 MVP와 백엔드 계약이 확정된 뒤 시작한다.
- React Native/Expo 기반의 별도 모바일 클라이언트를 구성한다.
- 현재 서버의 HTTP API와 Socket.IO를 사용하고 PostgreSQL에 직접 연결하지 않는다.
- 모바일 토큰 갱신, 안전한 자격 증명 저장, 앱 생명주기, 재연결을 구현한다.
- 오프라인 조회 캐시와 멱등 키를 유지하는 미전송 메시지 대기열을 구현한다.
- 푸시 알림, 딥 링크, 앱 배포 구성을 추가한다.

## 우선순위

1. Socket 인증과 공용 권한 계층
2. Redis와 다중 서버 실시간 상태
3. 공용 API·이벤트 계약
4. 운영 배포·백업·로그
5. 웹 MVP 범위 확정 및 버전 태그
6. 같은 백엔드를 사용하는 모바일 앱

## 현재 다음 작업

Socket.IO 연결과 `user:join` 흐름에서 사용자 신원을 서버가 발급한 티켓으로 검증하도록 설계하고, 모든 채팅 이벤트에 공용 방 권한 검사를 적용한다.

## 이번 단계에서 하지 않는 작업

- PostgreSQL을 Redis나 다른 데이터베이스로 교체
- Kafka 도입
- 마이크로서비스 분리
- PostgreSQL 샤딩
- Kubernetes 도입
- 사용량 측정 없이 메시지 테이블 파티셔닝 적용
